3.2.2. Data parsing

3.2.2.1. Events

We parsed the individual html files from step Section 3.2.1.4 to extract raw event data. An event is an instance of work or education attributed to an employee. In its raw form, each event has the following information:

  • The person that does the event

  • Start / end date

  • The organization where the person worked or studied (name and unique id)

  • The role or job title (for work events) or the education program (for study events)

  • A role or program description

  • A location (a city, region, country, etc.)

Filling in all fields is not mandatory, so some field values are missing in events.

3.2.2.2. Locations

We processed locations in two steps:

  1. we harmonized location data by canonicalizing, cleaning and parsing the raw location strings using various Python NLP modules

  2. we looked up the “clean” location strings using geocoder, and obtained their coordinates (latitude and longitude)

  3. we looked up the coordinates using geopy to obtain disambiguated location meta-data (location name, country, ISO-3166-2 codes, etc.)

Table location includes concise location data for each unique location uuid. Table location_link relates raw and parsed locations, and includes fine grained location data.

Example: parsing locations in Brussels

Our algorithm to parse location maps the following location strings

  • Parvis Sainte-Gudule, 1000, Bruxelles

  • Brussels

  • Rue de la Loi 42, 1040, Bruxelles

  • Bourgetlaan 60, 1140, Brussel

to Ville de Bruxelles - Stad Brussel (ISO-3166-2 label: “BE-BRU”)

3.2.2.3. Organization names

When a person creates an event in WEFP, either work experience or education, they can list the organization where this work event took/is taking place. If WEFP has a dedicated page for the organization, a link to it is provided in the event. This link is a URL which contains the organization’s name and unique numeric identifier. Hence, by parsing organization information from the events we generate a new mapping of organization names and their unique numeric id. For example, if a person creates a study event that includes KU Leuven, while displaying the event WEFP will show a URL containing both Katholieke Universiteit Leuven and 3629 (KU Leuven’s unique numeric id at WEFP).

Note

Organization names are entered by WEFP users and only lightly harmonized by WEFP. Hence, both Katholieke Universiteit Leuven and KU Leuven (and many other variants, including spelling mistakes as well) may appear in events where users worked or studied at this university.

WEFP does not fully harmonize the organization name entered by the user, and this is the organization name displayed in the event. However, WEFP performs a matching task in the background in order to match user-entered organization names to organizations’ unique WEFP pages. This is why organizations with somewhat different names in the front-end are linked to the same organization in the back-end (i.e., all these KU Leuven spelling variations are linked to the unique id 3629).

While useful, this step may introduce errors, if the match done by WEFP is incorrect. For instance, an employee of Datacamp Brazil (a software vendor) may be linked to Datacamp (the more prominent online education platform).

However, not all organizations in the database have a dedicated page at WEFP. When organizations have a dedicated page at WEFP, we use names, unique numeric id, and locations to disambiguate organization data; when they do not, we use only names and location data. We proceeded as follows:

  1. we harmonized organization names data by canonicalizing, cleaning and parsing the raw organization name strings using various Python NLP modules

  2. we combined the “clean” organization names with the WEFP organization numeric id’s and disambiguated location data to generate disambiguated organization uuid’s

Table organization contains the organization names and WEFP numeric ids, whereas a row in table organization_link connects raw and parsed organization names.

3.2.2.4. Job titles

We processed job title (also referred as job role) strings to harmonize them, but also to extract from them the job’s functional area (e.g., accounting, marketing, HR, etc.) and rank (e.g., top management, mid-management, non-management). We proceeded as follows:

  1. we harmonized job titles by canonicalizing, cleaning and parsing the raw job title strings using various Python NLP modules

  2. we used TextBlob to correct spelling mistakes

  3. when a single job title string contained two or more comma-separated parts (e.g., “.NET developer, team leader”; “accounts executive, internship”) we processed each part separately

  4. to deal with job titles in languages other than English we used polyglot to detect their language, and translate-api to get their English translation

  5. we matched the clean, spell-checked, and translated job titles against the “Alternate [job] titles” database provided by the O*NET project. This helped us to further disambiguate the job titles, but also to match them to a rich external database with job skills, tasks, ranks, etc.

Note

We keep track of the parsing steps in role_link and program_link. The first column shows the raw string, whereas the columns appended with _split, _spellcheck, etc, show (from left to right) the parsing process.

3.2.2.4.1. Extracting job ranks and functional areas

We followed the methodology in Chen and Thompson [CT16] to classify the parsed job titles into:

  • Job ranks: top management, management, sub-management and non-management

  • Functional areas: Business and Management, Production, R&D and Engineering, Personnel, Sales and Marketing, Accounting and Finance

  • Other indicators (e.g., engineering role, medical role, etc.)

The cited methodology uses sets of keywords linked to each of the fields above to assign a job title string to one class if two conditions are fulfilled:

  • the string contains one or more keywords linked to a class,

  • the string does not contain a keyword in a set of exclusions, particular to that class.

For instance, we classify job title strings containing terms such as sales, key account or marketer into Sales and Marketing. However, we abstain from doing so if the string also contains a term such as bond (e.g., in bond sales).

Note

We used the dictionary provided by Chen and Thompson [CT16] and expanded it with new terms after careful examination of the job title string in our database.

3.2.2.5. Study programs

We parsed the edu_prg raw string in order to identify

  • the level of education of each study program (primary school, secondary school, bachelor, etc.), and

  • the field of each study program (Arts, humanities, ICT, etc.).

We proceeded as follows:

  1. we harmonized program titles by canonicalizing, cleaning and parsing the raw strings using various Python NLP modules

  2. we used TextBlob to correct spelling mistakes

  3. when a single program title string contained two or more comma-separated parts (e.g., “1998, Mécanique, Electronique”; “Algemeen Secundair Onderwijs, Economie Moderne Talen”) we processed each part separately

  4. to deal with program titles in languages other than English we used polyglot to detect their language, and translate-api to get their English translation

  5. we developed tools to detect, clean and harmonize the parts of the string that refer to the level of the study program (e.g., ‘master’ -> ‘msc’, ‘Ph. d.’ -> ‘phd’)

Note

Please note that while we applied several NLP steps and tools to study programs, we never aimed to fully disambiguate them. Rather we fixed spelling mistakes, reduced string variation lightly (e.g., engineering -> enginner), translated strings, and targeted the parts of the string that indicate a level (e.g., primary, secondary, master).

3.2.2.6. Age

We use disambiguated schooling data to estimate entrepreneurs’ birth years and derive the age at the time of work experience, education and cofounding. In this step we combine information from the study level and the starting or ending year of each entry in the education table. First, for each education entry with ISCED level and at least start or end year of education program, we used the mapping rule in table Table 3.2 to estimate the birth year of the entrepreneur. For example, if Jane started secondary school in 2005, we estimate she was 12 at the time, and impute a birth year of 1993. Second, if an entrepreneur has many education entries in the education table, this step may produce different estimates of birth year. In such case, we take the minimum of all estimated birth years of each entrepreneur. Third, we combined the (minimum) estimated birth year and the cofounding year to estimate the age of the entrepreneur at the moment of cofounding a firm.

3.2.2.7. Gender

We used gender-guesser on the raw person names to guess the gender of the employees. The package takes a person name and classifies is into one of five categories: male, female, mostly_male, mostly_female, andy (androgynous, when the name has the same probability to be male than to be female), and unknown.

We reclassified “mostly_female” as “female” and “mostly_male” as “male”.

Note

Our current algorithm cannot guess the gender of about 9% of the person names, we are working to improve the coverage in future versions. Please have in mind this figure if you undertake any analysis that relies on the gender classification.
